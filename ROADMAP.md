# Дорожная карта XimFinance

## Текущее состояние

Приложение включает следующие разделы: Главная, Себестоимость, Транзакции, Контрагенты, Категории, Отчёты.

Стек: React + TypeScript (фронтенд), Node.js + Express + TypeORM (бэкенд), PostgreSQL через Supabase.

---

## Этап 1 — Раздел «Экономика»

> Цель: объединить расчёт себестоимости, UNIT-экономику и расчёт маржинальности в единый раздел с навигацией по подразделам.

### 1.1 Себестоимость

Перенести существующую страницу `CostCalculation` под новый раздел «Экономика». Логика и данные не меняются.

### 1.2 UNIT-экономика

Расчёт на одну единицу товара. Формула:

```
UNIT-маржа = Цена продажи
           − Себестоимость
           − Логистика
           − Хранение
           − Рекламный бюджет (ДРР, % от выручки)
```

Параметры для ввода:

| Параметр | Описание |
|---|---|
| Себестоимость | Берётся из раздела «Себестоимость» (авто или ручной ввод) |
| Логистика | Стоимость доставки до покупателя, руб/шт |
| Хранение | Стоимость хранения за период, руб/шт |
| ДРР (%) | Доля рекламных расходов от выручки |
| Цена продажи | Розничная цена на канале |

### 1.3 Расчёт маржинальности

Расчёт ведётся по трём каналам продаж: **Wildberries**, **Ozon**, **Сайт**. Каждый канал имеет собственные параметры UNIT-экономики, так как логистика, хранение и ДРР различаются.

Итоговая таблица маржинальности:

| Показатель | WB | Ozon | Сайт |
|---|---|---|---|
| Цена продажи | — | — | — |
| Себестоимость | — | — | — |
| Логистика | — | — | — |
| Хранение | — | — | — |
| Реклама (ДРР) | — | — | — |
| **Маржа, руб** | — | — | — |
| **Маржинальность, %** | — | — | — |

Новые сущности бэкенда:

- `SalesChannel` — канал продаж (WB, Ozon, Сайт), настраиваемые параметры
- `UnitEconomics` — сохранённые расчёты UNIT-экономики по каналу и периоду
- `MarginCalculation` — итоговый расчёт маржинальности (связь с UnitEconomics)

Новые API-маршруты:

```
GET    /api/economics/channels
POST   /api/economics/channels
GET    /api/economics/unit
POST   /api/economics/unit/calculate
GET    /api/economics/margin
POST   /api/economics/margin/calculate
```

## Этап 2 — Экспорт данных в XLS

> Цель: дать возможность выгрузить любой отчёт или таблицу в файл Excel.

Библиотека: `exceljs` на бэкенде (поддерживает форматирование, заголовки, несколько листов).

Разделы с поддержкой экспорта:

- Транзакции (с текущими фильтрами)
- Отчёты по категориям и контрагентам
- Расчёт маржинальности (по каналам)
- БДДС (план vs факт)
- P&L

API:

```
GET /api/export/transactions?startDate=&endDate=
GET /api/export/reports?type=category|counterparty&startDate=&endDate=
GET /api/export/margin?channelId=
GET /api/export/bdds?year=
GET /api/export/pl?year=
```

Ответ: файл `.xlsx` с заголовком `Content-Disposition: attachment`.

На фронтенде — кнопка «Экспорт в XLS» в заголовке каждого соответствующего раздела.

## Этап 3 — Раздел «Документы»

> Цель: хранить и систематизировать документы от контрагентов и подрядчиков.

### Функциональность

- Загрузка файлов (PDF, DOCX, XLSX, изображения)
- Привязка к контрагенту
- Типы документов: договор, акт, счёт, счёт-фактура, УПД, прочее
- Фильтрация по типу, контрагенту, дате, статусу
- Просмотр и скачивание файла
- Статус документа: черновик, на подписании, подписан, архив

### Хранилище

Файлы загружаются в **Supabase Storage** (bucket `documents`). В базе хранятся только метаданные.

Новая сущность `Document`:

| Поле | Тип | Описание |
|---|---|---|
| id | uuid | — |
| title | varchar | Название документа |
| type | varchar | Тип (договор, акт и др.) |
| counterparty_id | uuid | Контрагент |
| status | varchar | Статус |
| file_url | text | URL в Supabase Storage |
| file_name | varchar | Оригинальное имя файла |
| file_size | int | Размер в байтах |
| document_date | date | Дата документа |
| notes | text | Примечания |

API:

```
GET    /api/documents
POST   /api/documents          (multipart/form-data)
GET    /api/documents/:id
PATCH  /api/documents/:id
DELETE /api/documents/:id
```

## Этап 4 — Развитие раздела «Категории»

> Цель: превратить категории из простого справочника в полноценный инструмент управления бюджетом.

### Улучшения

**Иерархия категорий**
- Поддержка родительской категории → подкатегории (1 уровень вложенности)
- Пример: «Маркетинг» → «Реклама WB», «Реклама Ozon», «SMM»

**Привязка к бюджетным статьям**
- Каждая категория привязывается к статье БДДС и/или строке P&L
- Используется для автоматического формирования отчётов

**Статистика по категории**
- Оборот за выбранный период (доход / расход)
- Доля категории в общих расходах / доходах, %
- График динамики по месяцам
- Количество транзакций

**Управление**
- Объединение категорий (merge)
- Деактивация без удаления
- Массовое переназначение транзакций при удалении категории

## Этап 5 — Отчётность: БДДС и P&L

> Цель: добавить управленческую отчётность для контроля финансового состояния бизнеса.

### 5.1 БДДС (Бюджет движения денежных средств)

Таблица план/факт по месяцам в разрезе статей (категорий).

Структура:

```
БДДС
├── Поступления
│   ├── Выручка WB
│   ├── Выручка Ozon
│   └── Выручка Сайт
└── Выбытия
    ├── Себестоимость
    ├── Логистика
    ├── Реклама
    ├── Зарплата
    └── Прочие расходы
```

Функции:
- Ввод плановых значений вручную по статьям на месяц
- Факт подтягивается автоматически из транзакций
- Отклонение = Факт − План, подсвечивается цветом
- Экспорт в XLS

### 5.2 P&L (Profit & Loss)

Отчёт о прибылях и убытках за период.

Структура:

```
P&L
├── Выручка (Revenue)
├── − Себестоимость (COGS)
├── = Валовая прибыль (Gross Profit)
├── − Операционные расходы (OpEx)
│   ├── Маркетинг
│   ├── Логистика
│   └── Прочие
├── = EBITDA
├── − Амортизация (если применимо)
└── = Чистая прибыль (Net Profit)
```

Функции:
- Автоматическое формирование из транзакций по привязанным категориям
- Сравнение периодов (текущий месяц vs предыдущий, текущий год vs предыдущий)
- Экспорт в XLS

Новые сущности бэкенда:
- `BudgetPlan` — плановые значения БДДС (период, статья, сумма)
- `PLItem` — строки P&L с привязкой к категориям

Новые API-маршруты:

```
GET  /api/reports/bdds?year=
POST /api/reports/bdds/plan
GET  /api/reports/pl?startDate=&endDate=
```

## Этап 6 — BI-визуализация

> Цель: расширить аналитические возможности дашборда и/или подключить внешний BI-инструмент.

### Варианты реализации

**Вариант A — Расширение встроенного дашборда (Recharts)**

Recharts уже используется в проекте (`frontend/src/pages/Reports.tsx`).

Доработки:
- Фильтры по периоду, каналу продаж, категории
- Drill-down: клик по категории → список транзакций
- Новые графики: водопадная диаграмма P&L, тепловая карта расходов по месяцам
- Сравнение периодов на одном графике

**Вариант B — Metabase (рекомендуется как дополнение)**

- Самостоятельно подключается к Supabase PostgreSQL
- Не требует изменений бэкенда
- Готовые дашборды, фильтры, drill-down
- Возможна iframe-интеграция в приложение

**Вариант C — Grafana**

- Подходит для операционного мониторинга
- Требует PostgreSQL datasource
- Менее удобен для финансовых отчётов

### Рекомендация

На первом этапе — расширить встроенный дашборд (Вариант A). Параллельно поднять Metabase (Вариант B) для ad-hoc аналитики и тяжёлых отчётов.
