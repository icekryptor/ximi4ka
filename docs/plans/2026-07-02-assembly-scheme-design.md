# Схема сборки наборов — Design

**Date:** 2026-07-02
**Scope:** Визуализация производственной схемы сборки (коллапсибл-дерево справа-налево), карточки операций с нормативами и себестоимостью работы, база знаний с регламентами. Наборы: Химичка 3.0 + Электрохимичка.

## Goal

Производственный процесс (10 этапов: розлив → закупорка → фасовка → ложементы → коробочки → дно → печатка → закрытие → защитный короб) существует только в голове оператора и частично в `component_parts`. Нужна страница, где дерево сборки видно целиком, себестоимость каждой сущности (материалы + работа) считается автоматически, а к каждому шагу привязан регламент и норматив времени.

## Решение: подход A — расширение BOM-модели

Дерево уже живёт в `component_parts` (композит → части). Добавляем слой **операций поверх композитов**, не дублируя дерево.

**Ключевое решение — операция привязана к композиту, не к ребру.**
У одного узла может быть несколько операций (розлив + закупорка обе производят «готовый флакон»). Клик на ребро открывает карточку целевого композита со списком его операций. Модель «операция-на-ребре» отвергнута: розлив соединяет 4 ребра (вещество, вода, флакон, этикетка), но работа одна — иначе двойной счёт и ×4 заполнение нормативов.

Отвергнуто также: react-flow graph-редактор с drag-n-drop (v2, YAGNI — состав правится существующим parts-CRUD).

## Данные

```sql
CREATE TABLE assembly_operations (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  composite_id uuid NOT NULL REFERENCES components(id) ON DELETE CASCADE,
  name varchar(255) NOT NULL,          -- «Розлив», «Закупорка и проклейка»
  stage int NOT NULL DEFAULT 0,        -- этап 1-10
  time_seconds int,                    -- норматив (NULL = не заполнен)
  instruction_slug varchar(100),       -- слаг регламента в brand_docs (kb-*)
  sort_order int NOT NULL DEFAULT 0,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);
CREATE INDEX idx_assembly_ops_composite ON assembly_operations (composite_id);

CREATE TABLE app_settings (
  key varchar(100) PRIMARY KEY,
  value jsonb NOT NULL,
  updated_at timestamptz DEFAULT now()
);
INSERT INTO app_settings VALUES ('labor_rate_per_hour', '500');
```

Стоимость работы операции = `time_seconds / 3600 × ставка` — на лету, не хранится. Смена ставки пересчитывает всё.

**Себестоимость узла (rollup):**
- лист (не композит): `reagent/metal → qty × price_per_gram`, иначе `qty × unit_price`
- композит: `Σ rollup(часть) × qty + Σ работы своих операций`

## Каркас операций (создаётся сразу, нормативы NULL — заполняются через UI)

Этапы: 1 розлив, 2 закупорка/проклейка, 3 фасовка, 4 ложементы флаконов, 5 ложементы пробирок, 6 коробочки, 7 сборка дна, 8 печатка, 9 закрытие, 10 защитный короб.

- Каждый раствор (19): «Розлив» (stage 1) + «Закупорка и проклейка» (stage 2)
- Каждая фасовка (капсула/баночка/пакетик): «Фасовка» (stage 3)
- Ложементы (флаконы Х №1, №2, ЭХ; пробирки Х, ЭХ): «Наполнение» (stage 4/5)
- Коробочки (лаб Х, лаб ЭХ, тв. реактивы ЭХ): «Наполнение» (stage 6)
- Дно Х/ЭХ (stage 7), печатка (stage 8), закрытие (stage 9), защитный короб (stage 10)

Итого ~70 заготовок. Ставка труда 500 ₽/час (app_settings).

## Достройка деревьев

**Химичка 3.0** (почти совпадает с реальностью):
- Вставить промежуточный узел «Дно + печатная продукция Х 3.0» (дно + методичка + листовки), «Собранный набор» = этот узел + крышка (разделение этапов 8/9)
- Верхний узел «Набор в защитном коробе Х 3.0» = Готовый набор + Защитная коробка (+ скотч/стикер, материалы добавятся позже)

**ЭХ** (с нуля):
- Ложемент флаконов ЭХ (новый компонент, цена 0 — внесётся) + 6 растворов: гидросульфат натрия, дихромат калия 6%, иодид калия 3%, NaOH, серная, хлорид железа III
- Ложемент пробирок ЭХ (чёрный) + 6 пробирок
- Коробочка тв. реактивов ЭХ = баночка S (чёрная) + баночка SnCl₂ (белая) + баночка KIO₃ (белая) + пакетики Mg/Fe-вата/Cu + капсула с цинком
- Коробочка лаб. оборудования ЭХ = мелкое оборудование из листа (разумная гипотеза, правится parts-CRUD)
- Дно ЭХ → + печатка → + крышка → готовый набор → защитный короб

**Пересборка фасовок ЭХ** (по уточнению): «Капсула с хлоридом олова» → «Баночка с хлоридом олова» (белая), «Капсула с иодатом калия» → «Баночка с иодатом калия» (белая), сера → «Баночка с серой» (чёрная), «Пакетик с магнием», «Пакетик с железной ватой», «Пакетик с медью». Цинк ЭХ — существующая «Капсула с цинком».

## API

- `GET /api/assembly/tree?root=<componentId>` — всё дерево одним ответом: рекурсивная сборка на сервере из 3 batch-запросов (components, component_parts, assembly_operations), rollup посчитан. ~100 узлов, без кэша.
- `GET /api/assembly/roots` — список корневых наборов (компоненты «Набор в защитном коробе *»)
- CRUD операций: `POST/PUT/DELETE /api/assembly/operations(/:id)` (инлайн-редактирование норматива из карточки)
- `GET/PUT /api/assembly/settings/labor-rate`
- База знаний — существующие brand_docs endpoints (слаги `kb-*`)

## Frontend

**Страница «Схема сборки»** (`/production/assembly`, сайдбар Производство):
- Селектор набора сверху
- Дерево справа-налево: справа конечный узел, ➕ раскрывает части слева. Узел = название + себестоимость (материалы+работа) + чип этапа. Рекурсивный React-компонент, линии SVG, без graph-библиотек
- Клик на ребро/узел → правая панель: операции узла (название, норматив — инлайн-редактор, стоимость работы, ссылка на регламент), состав с ценами
- Состояние раскрытия — локальный useState (Set of ids)

**Страница «База знаний»** (`/production/knowledge-base`):
- Список документов brand_docs со слагом `kb-*` + markdown-редактор (переиспользование готового редактора стратегии)
- Из карточки операции — селектор регламента + «открыть»

## Failure handling

| Сбой | Поведение |
|---|---|
| Цикл в component_parts | Backend-обход помечает посещённые узлы; цикл обрубается с warning в ответе |
| time_seconds NULL | Работа = 0, в карточке бейдж «норматив не заполнен» |
| instruction_slug → удалённый док | Карточка показывает «регламент не найден», ссылка на КБ |
| Компонент без цены (ложемент ЭХ) | Материалы = 0, узел помечается «⚠ нет цены» |
| Ставка не задана | Дефолт 500, warning в консоль |

## Testing

Typecheck backend/frontend. Smoke: дерево Х 3.0 и ЭХ открывается, rollup совпадает с ручным расчётом контрольного узла (раствор сульфата меди = 8.05 ₽ + работа), инлайн-редактирование норматива меняет стоимость, регламент создаётся в КБ и открывается из карточки. E2E по обоим наборам через Playwright.
