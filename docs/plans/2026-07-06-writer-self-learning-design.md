# Самообучение Writer'а (волна 2) — Design

**Date:** 2026-07-06
**Scope:** Цикл самообучения агента-копирайтера: правка копирайтера → анализ различий в Cowork → извлечение паттернов-правил → сохранение в БД через MCP → следующие генерации читают накопленные правила. Отражение петли в харнессе. MCP-native (ноль вызовов Claude из ERP).

## Goal

Writer пишет статично. Нужно, чтобы он **учился на правках**: копирайтер исправил текст → агент понял, ЧТО и ПОЧЕМУ изменилось, вывел общее правило, дописал его в свою инструкцию, и следующий текст уже учитывает это. Накопление правил приближает стиль к авторскому за ~10-15 итераций на формат. Сигнал «обучен»: 2-3 итерации подряд дают 0 новых правил.

## Механика (4 шага пользователя)

1. Writer генерит текст (в Cowork).
2. Копирайтер правит, приносит агенту оригинал + правку (или заметки).
3. Агент в Cowork анализирует разницу → извлекает паттерны → сохраняет через MCP-инструмент.
4. Агент перерабатывает текст, применяя обновлённый свод правил.

ERP = хранилище правил + генератор промпта-анализа. LLM-работу делает агент в Cowork. Формат правила — как в существующем `editWithLearning`: `{ code, title, before, after, rationale }`, категории А11+ (аудио/ритм), С10+ (структура/лексика), Э8+ (узкие уточнения).

## Данные

**Таблица `style_pattern`** (гибрид-индекс правил):
```sql
CREATE TABLE style_pattern (
  id uuid PK,
  format varchar(50) NOT NULL,     -- content_type (short_post, ...)
  code varchar(16) NOT NULL,       -- А11 | С10 | Э8 ...
  title varchar(255) NOT NULL,
  before text,                     -- «как НЕ надо»
  after text,                      -- «как надо»
  rationale text NOT NULL,
  source_note text,                -- контекст правки (откуда правило)
  created_at timestamptz DEFAULT now()
);
CREATE INDEX idx_style_pattern_format ON style_pattern (format);
```
Entity + миграция + регистрация в `allEntities`.

Почему таблица, а не brand_doc: правила структурированы (нужны code/before/after), считаемы (сигнал сходимости = число новых за период), группируемы по формату, показываемы в харнессе.

## Backend

**`content-context.service.ts` (дополнить):**
- `listStylePatterns(format): Promise<StylePattern[]>` — накопленные правила формата, сортировка по code.
- `saveStylePatterns(format, patterns[], sourceNote?): Promise<{added:number; skipped:number}>` — вставка; дедуп по (format, code) — не плодить дубли; вернуть счётчики (added=0 → сигнал сходимости).
- `buildLearningPrompt({format, original, edited?, notes?}): Promise<string>` — собрать промпт-анализа: системная инструкция из `editWithLearning` (та же логика извлечения А/С/Э) + ТЕКУЩИЕ правила формата (чтобы агент не дублировал) + оригинал/правка. Чистая строка, без callClaude.
- `getWriterContext`: добавить блок «## Накопленные правила стиля (применяй)» из `listStylePatterns(format)` — Writer применяет их при генерации. reads += маркер `style_learned`.

**MCP-инструменты (дополнить `content-mcp.server.ts`):**
- `learn_from_edit({ format, original, edited?, notes? })` → возвращает промпт-анализа (buildLearningPrompt). Агент выполняет анализ в Cowork.
- `save_style_patterns({ format, patterns:[{code,title,before,after?,rationale}], source_note? })` → saveStylePatterns; ответ «добавлено N, пропущено M (дубли). N=0 → стиль стабилизировался».
- `list_style_patterns({ format })` → накопленные правила (агент для дедупа + харнесс).

Итого 12 MCP-инструментов (было 9 + 3).

## Frontend — отражение в харнессе

**Writer-шаг:** новая веточка reads `style_learned` — «Накопленные правила стиля (N)». Клик → панель с правилами формата (code, title, before→after, rationale), сгруппировано; сигнал сходимости («+0 новых за последние N правок → стиль пойман»).

**CoworkGuide:** добавить блок «🔄 Обучение Writer'а»:
1. Дал агенту оригинал + правку → он зовёт `learn_from_edit`.
2. Анализирует разницу, извлекает правила (не опечатки, не разовое).
3. Зовёт `save_style_patterns` — правила сохранены, применятся впредь.
4. Переписывает текст по обновлённому своду.
Пометка про сходимость (~10-15 итераций, сигнал = 0 новых).

**Секция «Обучение стиля»** (компактная, под контент-планом): таблица правил по формату + счётчик + дата последнего. Read-only (правила пишет агент через MCP; ручное удаление опционально v3).

## Failure handling

| Ситуация | Поведение |
|---|---|
| Правил для формата нет | writer_context без блока правил; харнесс «правил пока нет» |
| save с дублирующим code | дедуп, skipped++, added не растёт |
| learn_from_edit без original | ошибка «нужен original» |
| паттерн-мусор (агент выдумал) | ответственность агента; инструкция «0 лучше мусора» (как в editWithLearning) |

## Testing

Typecheck. Локальный MCP: `list_style_patterns` (пусто) → `save_style_patterns` (added=2) → `list_style_patterns` (2) → `save` те же коды (skipped=2, added=0 = сигнал) → `writer_context` содержит блок правил. `learn_from_edit` возвращает промпт с текущими правилами. Прод-smoke. Playwright: харнесс показывает веточку style_learned + секцию правил, гайд обновлён.
